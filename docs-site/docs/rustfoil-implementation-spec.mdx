# Faithful Solver Reference Index

> **Document Status**: Current reference  
> **Focus**: Faithful viscous operating-point solver  
> **Last Updated**: 2026-03-14

This is the entry point for the current solver documentation set. It replaces the earlier implementation-plan framing with a reference organized around the solver that now acts as the faithful XFOIL recreation.

The documentation here is based on:

- the live faithful path in `crates/rustfoil-xfoil`
- the shared `rustfoil-coupling` and `rustfoil-bl` kernels used by that path
- XFOIL Fortran sources in `Xfoil-instrumented/src`
- recent Obsidian workstreams and daily notes covering the parity work from 2026-03-10 through 2026-03-13

## Per-Function Template

The detailed pages use the same structure for each documented function:

- `Function`: Rust symbol and source file
- `Role`: what it does in the faithful loop
- `XFOIL Mapping`: direct routine, partial analogue, or Rust-only support layer
- `Fortran Reference`: file and routine location, with a short excerpt where useful
- `Why This Looks Different`: the main Rust-side adaptation or design choice
- `Current Fidelity Status`: faithful, mostly faithful, or still transitional

## Active Architectural Notes

These themes appear repeatedly in the detailed pages because they came directly out of the recent parity work:

- preserve XFOIL mutation order even when Rust abstraction would be cleaner
- keep sign conventions explicit on the lower surface and in wake coupling
- prefer one canonical owner state, but allow typed kernel adapters where they improve clarity without changing behavior
- document any routine that is still acting as a bridge rather than a literal one-to-one port

## Status Legend

| Status | Meaning |
| --- | --- |
| `faithful` | The function is intended to mirror a named XFOIL routine closely in call role and behavior. |
| `mostly faithful` | The function tracks the XFOIL routine closely but still reflects Rust-side factoring or current parity work. |
| `support layer` | The function exists to support the faithful path, but it is not a direct named XFOIL routine. |
| `known gap` | The function is on the faithful path, but the docs currently record an active parity or ownership caveat. |
